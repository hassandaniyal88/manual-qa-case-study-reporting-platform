Building a Scalable Multi-Agency Reporting System: A QA-Led Case Study

INTRODUCTION

In 2025–2026, I was responsible for QA on one of the most complex modules of a national-scale public safety platform. The platform enables civilians to submit incident reports that must be routed to the correct public safety agency anywhere in the United States. To achieve this, the system uses a multi-workspace architecture, where each agency operates in its own isolated workspace with custom rules, users, and activation states.

My role as QA was not only to test functionality, but also to analyze requirements, identify logic gaps, validate user flows, write clarification questions, and ensure that all roles, workspaces, and workflows behaved consistently across the entire system.

This case study documents how I, as a QA Engineer, helped build and validate a scalable multi-agency system that supports nationwide onboarding.

BACKGROUND

The system supports four primary user roles, each with different permissions:

  • Civilian Users
  • Agency Staff
  • Supervisory Users
  • System Admins

Each agency has its own workspace containing:

  • Its own users
  • Its own reports
  • Its own public content
  • Its own activation status (Draft → Pending → Live)

The challenge was to implement an engine that allows a civilian anywhere in the country to:

  • Search and join their local agency
  • Switch agencies if traveling
  • Submit reports with correct routing
  • See public information in a statewide, role-filtered format
  • Follow strict logic rules that vary by user type

This required deep testing, requirement discovery, and logic refinement.

THE CHALLENGE

The initial documentation for this module had several issues:

1. Requirements were incomplete or contradictory

Multiple documents had different expectations for:

  • Agency activation states
  • Cross-agency visibility
  • Report routing
  • Public post visibility
  • Subscription/contract workflows

2. Many real-world scenarios were not defined

  • Civilian lives in one city but reports an incident in another
  • An agency is visible but not accepting reports
  • Public content overlaps between agencies
  • Multi-step subscription logic with many states

3. Complex multi-role behavior

Rules applied differently depending on role, workspace state, or report type.

4. No single source of truth

Implementation spanned multiple documents, emails, UI files, and refinement notes.

MY ROLE AS QA ENGINEER

I took a leadership role in defining clarity, structure, and consistency. My responsibilities included:

  • Requirements Analysis — cross-checking all major documents and identifying missing logic.
  • Logic Gap Detection — discovering undefined or conflicting behaviors.
  • Clarification Questions — validating assumptions with the client early.
  • Full QA Planning — designing detailed scenarios across roles, states, and workflows.
  • Defect Identification — testing and finding system-level issues.
  • Ticket Creation — converting vague requirements into dev-ready acceptance criteria.
  • Maintaining Feedback Tracking — ensuring alignment between QA, dev, and product.

KEY LOGIC GAPS I DISCOVERED

Gap 1 — Cross-agency visibility rules were inconsistent

Should civilians see:

  • Only their agency’s content?
  • Nearby agencies?
  • Full statewide feed?

After QA-led clarification:

  • User’s Primary Agency
  • Nearby Agencies (based on configuration)
  • Statewide Content

Gap 2 — Undefined agency compatibility for report submissions

Civilians could enter an incident address that didn’t match the selected agency.

After my QA inquiry: A compatibility fail-safe modal was introduced.

Gap 3 — Activation state behavior was unclear

Rules for Draft, Pending, and Live states were not defined.

Final clarification:

  • Draft → not public
  • Pending → selectable but cannot receive reports
  • Live → fully active

Gap 4 — Civilian data persistence not defined

Clarified: Civilian data belongs to the user, not the agency.

Gap 5 — Subscription lifecycle missing contract instance rules

Defined the rule: each agency must have one active contract instance.

QA SOLUTIONS & CONTRIBUTIONS

  • Created a unified logic model
  • Developed 60+ structured test cases
  • Merged conflicting requirements into one truth source
  • Improved developer clarity with actionable tickets
  • Strengthened client communication
  • Validated scalability for nationwide onboarding

OUTCOME & IMPACT

  • Zero misrouted reports after compatibility fail-safe
  • Faster agency onboarding due to clear activation logic
  • Architecture stability through clarified requirements
  • Reduced rework via precise QA-driven logic
  • Consistent UX across all user roles

WHAT I LEARNED

  • QA is not just testing — it’s product thinking
  • Clarification early prevents weeks of rework
  • Complex systems require structured decision trees
  • Documentation must match real behavior
  • QA protects users, developers, and the business

CONCLUSION

Building a scalable multi-agency reporting engine requires more than code — it requires clear logic, deep testing, and proactive QA leadership. Through systematic analysis, continuous clarification, and structured testing, I ensured the system could scale nationwide while maintaining accuracy, reliability, and usability.
